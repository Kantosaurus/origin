# Gaps: origin vs. peer coding agents

This document compares **origin** — a Rust-native agentic coding harness (thin CLI client + supervised background daemon + terminal TUI) — against eight peer coding-agent projects: **aider**, **claude-code**, **cline**, **gemini-cli**, **jcode**, **kilocode**, **openclaude**, and **opencode**. Each peer's repository (or, for closed-source claude-code, its public docs/plugins/changelog) was read on 2026-05-30, and for each project this report records only (a) what that project has that origin's documented baseline **lacks**, and (b) what it does **demonstrably better** than origin. Capabilities origin already matches are not listed as gaps.

**How to read this.** These are gaps and relative strengths, not a verdict on origin's quality. origin is pre-1.0 (0.0.1) and intentionally narrow in surface area; several of its differentiators (below) are unmatched by any peer here. The point of the comparison is to make deliberate roadmap choices about which gaps are worth closing and which are out of scope by design.

### origin's core differentiators (the baseline)

- **Rust-native CLI + supervised daemon + TUI**, talking only over a local IPC channel (origin-ipc) using rkyv archived frames; remote IPC via QUIC + mTLS.
- **Performance as first-class CI gates** — cold start, keystroke-to-pixel latency, steady RSS, cache hit rate; a perf-gate job asserts read-only tasks complete in <=80ms.
- **Two-runtime daemon** — `current_thread` control plane pinned to one OS thread + multi_thread workers, with task-class scheduling.
- **Content-addressed storage (origin-cas)** — Hot LRU / Warm mmap / Cold zstd tiers that dedupe tool outputs, file reads, embeddings, memory, and code-graph nodes across turns/sessions/swarm workers.
- **Archived IR end-to-end (rkyv)** — one byte buffer through IPC, SQLite blobs, and ring buffers (~200ns validate).
- **Embedding-indexed lazy skills** — skills indexed into an HNSW graph, top-K materialized per turn.
- **Per-platform sandboxing** — landlock+seccomp+namespaces (Linux), sandbox-exec (macOS), AppContainer (Windows); permission-gated tools (a skill without Bash literally cannot shell out).
- **KeyVault secrets** — OS-native credential stores + age-encrypted fallback, `Secret<T>` redaction, CI lint rejecting raw secret bytes in logs.
- **Deterministic replay** (.origin-replay bundles); zero `unsafe` in surface crates; idempotent migration import from claude-code/jcode/opencode.
- **~40 providers** behind one catalog; code graph, memory, goal driver, planner, swarm workers, hooks, MCP, LSP client; npm/cargo-binstall/Homebrew/winget/AUR distribution.

## Cross-cutting gaps

Synthesizing across all eight peers, the same themes recur. Ranked roughly by how broadly they appear and how central they are to user-facing competitiveness:

1. **IDE / editor surface (highest priority).** Every peer except none has one: VS Code and/or JetBrains extensions (claude-code, cline, gemini-cli, kilocode, openclaude, opencode, jcode-desktop), the open **ACP protocol** to embed in Zed/Neovim/JetBrains/Xcode (cline, gemini-cli, opencode), and aider's editor-agnostic AI-comment watcher. origin's deliberate CLI+TUI-only stance is the single largest divergence from the field. If origin keeps no native extension, an ACP server is the lowest-effort, highest-reach way to appear inside many editors at once.

2. **Web/GUI, mobile, and chat surfaces.** Browser apps (aider Streamlit, cline Kanban, kilocode dashboard, opencode web/desktop, claude.ai), mobile apps (claude-code, cline, kilocode, jcode iOS, openclaude Android), and chat-platform bots (cline, kilocode, opencode Slack; claude-code/gemini-cli/kilocode/opencode GitHub/GitLab) dominate the peer set. origin reaches only the terminal. Remote **attach-from-any-client** (opencode `attach`, jcode gateway over Tailscale) is the architectural enabler origin's QUIC+mTLS daemon could plausibly support.

3. **Autonomy and orchestration modes.** Peers go well beyond fixed swarm workers: jcode's resource-aware **Ambient/OpenClaw** and **Overnight** runs, claude-code **dynamic workflows** and **Agent Teams**, cline **agent teams** + Kanban worktree-per-task, kilocode **Gastown** (Mayor/Polecat/Refinery), and gemini-cli's declarative tool-isolated subagents + **A2A** federation. origin has a goal driver and swarm but no proactive always-on mode, no named-teammate collaboration, and no agent-to-agent protocol.

4. **Scheduling, webhooks, and CI/VCS automation.** Cron/event triggers (claude-code `/schedule`+`/loop`, cline cron, kilocode Triggers, opencode cron) and PR/issue bots that open PRs and run multi-agent review (claude-code, gemini-cli, kilocode, opencode) are widespread. origin has no scheduler, no webhook entry, and no GitHub/GitLab automation surface.

5. **Marketplace / plugin ecosystem and live config interop.** Installable bundles and registries (claude-code marketplace, gemini-cli extension gallery, opencode/cline npm plugins, kilocode) plus **live consumption** of Claude Code config/skills (kilocode claudeCodeCompat, opencode `.claude` reading) outpace origin's one-shot idempotent import and its hooks/skills/MCP without a packaging or distribution model.

6. **Task-success benchmarking and observability.** origin gates on latency/RSS/cache-hit but not correctness. cline (cline-bench, pass@k/pass^k) and aider (polyglot + SWE-bench + public leaderboard) measure task success; gemini-cli, cline, and jcode export **OpenTelemetry / product telemetry** (with prebuilt dashboards and agent-time/pain metrics) that origin's internal-only metrics lack.

7. **Cost/usage transparency and model routing.** Per-turn USD cost (aider), per-category cost breakdowns (claude-code `/usage`, kilocode microdollar), and automatic/scored model routing (gemini-cli Pro/Flash + Gemma router, openclaude SmartRouter, kilocode Virtual Quota Fallback) are common. origin tracks cache-hit internally but surfaces no dollar cost or automatic routing.

8. **Multimodal and voice input.** Voice-to-code (aider, claude-code, jcode, kilocode) and image/PDF/sketch ingestion (aider, claude-code, gemini-cli) are increasingly headline features. origin's baseline lists no vision or audio modality.

9. **Interactive review, checkpoints, and rewind.** Shadow-git checkpoints with granular restore (cline, kilocode), conversation rewind with selective file revert (gemini-cli), and in-editor diff approve/reject loops (cline, kilocode, gemini-cli) give a safety/undo UX that origin's deterministic replay bundles do not match interactively. aider's git-as-undo is a simpler but proven alternative.

10. **Editing-strategy and post-edit pipelines.** aider's model-tuned edit-format matrix and auto-lint/auto-test repair loop, opencode's apply_patch multi-file primitive and ~25 auto-formatters, and the broad auto-installed LSP fleets (opencode 40+) are concrete mechanical advantages over origin's described editing tools.

Where origin already leads — Rust-native zero-`unsafe` surface, the rkyv archived-IR-everywhere pipeline, content-addressed cross-session dedupe (origin-cas), the two-runtime daemon with sub-80ms perf gates, per-platform sandboxing tied to permission-gated tools, and HNSW-indexed lazy skills — no peer here clearly matches the combination. The strategic question is which of the surface/ecosystem/autonomy gaps above are worth adopting without diluting that performance-and-safety core; an **ACP server** (theme 1), **attach-from-any-client remoting** (theme 2), and **task-success benchmarking + OTel export** (theme 6) are the highest-leverage, lowest-architectural-risk places to start.
